# AI-Investment-Assistant

Интеллектуальная веб-система для анализа акций компаний на основе рыночных данных, новостей и искусственного интеллекта.

Проект рассчитан на короткую разработку за 2 дня, поэтому основная цель - сделать рабочий MVP: пользователь вводит тикер компании, получает ключевые данные по акции, новости, краткую оценку их влияния и итоговый AI-анализ.

---

## 1. Цель проекта

Разработать веб-приложение, которое позволяет пользователю:

- искать компанию по биржевому тикеру;
- получать актуальную информацию об акции;
- просматривать краткую информацию о компании;
- видеть последние новости по компании;
- получать анализ влияния новостей;
- получать итоговую AI-аналитику в понятном формате.

---

## 2. MVP-функциональность

### 2.1. Поиск акции

Пользователь вводит тикер компании, например:

- `AAPL`
- `TSLA`
- `MSFT`

Система должна:

- принять тикер с фронтенда;
- привести тикер к единому формату, например к верхнему регистру;
- проверить, что тикер не пустой;
- отправить запрос на backend;
- вернуть пользователю результат или понятную ошибку.

### 2.2. Информация об акции

Для найденной компании отображается:

- название компании;
- тикер;
- текущая цена;
- изменение цены за день в процентах;
- валюта.

### 2.3. Информация о компании

Отображается краткое описание компании:

- сфера деятельности;
- основные продукты или услуги;
- общая позиция на рынке.

Если внешний источник не возвращает полноценное описание, допускается краткий fallback-текст.

### 2.4. Новости компании

Система отображает последние новости по компании.

Для каждой новости:

- заголовок;
- дата публикации;
- краткое содержание;
- ссылка на источник, если она доступна.

Для MVP достаточно выводить 3-5 новостей.

### 2.5. Анализ новостей

Для каждой новости система определяет примерное влияние на компанию:

- положительное;
- отрицательное;
- нейтральное.

Для каждой оценки нужно краткое объяснение причины.

На первом этапе допускается простая реализация через AI API или через базовые правила, если подключение AI API не успевает быть готово.

### 2.6. Итоговый AI-анализ

На основе данных об акции, информации о компании и новостей ИИ формирует итоговый отчет.

Отчет должен содержать:

- общую оценку ситуации;
- ключевые факторы влияния;
- риски;
- итоговый вывод.

Рекомендуемый объем: 100-200 слов.

Важно: результат не должен подаваться как финансовая рекомендация. Нужно явно указывать, что это аналитическая справка, а не совет к покупке или продаже.

## 3. Архитектура

```text
Frontend -> Backend -> Data API -> AI API -> Backend -> Frontend
```

### Frontend

Отвечает за:

- форму ввода тикера;
- отображение состояния загрузки;
- отображение ошибок;
- карточку акции;
- блок информации о компании;
- список новостей;
- блок итогового AI-анализа.

### Backend

Отвечает за:

- прием запроса от frontend;
- валидацию тикера;
- получение данных из внешних API;
- нормализацию данных;
- запрос к AI API;
- возврат готового JSON-ответа frontend.

### Data API

Источник рыночных данных и новостей.

Возможные варианты:

- Yahoo Finance;
- Alpha Vantage;
- Finnhub;
- MarketStack;
- другой доступный API.

Для MVP можно использовать один основной источник данных и один источник новостей.

### AI API

Используется для:

- анализа тональности новостей;
- формирования итогового отчета.

Если AI API временно недоступен, нужно предусмотреть fallback: вернуть базовый текст с предупреждением, что AI-анализ недоступен.

## 5. Делегирование задач на команду из 3 человек

### Участник 1: Backend и интеграции

Зона ответственности:

- настроить backend-проект;
- создать endpoint для анализа тикера;
- реализовать валидацию тикера;
- подключить API рыночных данных;
- подключить API новостей;
- собрать единый JSON-ответ для frontend;
- обработать ошибки внешних API.

Основной результат:

- рабочий backend endpoint, например `GET /api/analyze/:ticker`;
- корректный JSON-ответ;
- понятные ошибки для неправильного тикера или недоступного API.

### Участник 2: Frontend и пользовательский интерфейс

Зона ответственности:

- сделать основной экран приложения;
- добавить поле ввода тикера и кнопку поиска;
- реализовать состояния загрузки, ошибки и пустого результата;
- сверстать блоки с данными акции, компании, новостями и AI-анализом;
- подключить frontend к backend;
- привести интерфейс к аккуратному и понятному виду.

Основной результат:

- пользователь может ввести тикер и увидеть результат;
- интерфейс не ломается при ошибках;
- данные удобно читать.

### Участник 3: AI-анализ, тексты, тестирование и сборка

Зона ответственности:

- подготовить prompt для AI-анализа;
- реализовать или помочь реализовать анализ влияния новостей;
- проверить качество итогового текста;
- протестировать основные сценарии;
- подготовить демонстрационные тикеры;
- оформить README и финальное описание проекта;
- помочь со сборкой и финальной интеграцией.

Основной результат:

- понятный AI-отчет;
- список проверенных сценариев;
- финальная документация;
- готовность проекта к демонстрации.

## 6. План работы на 2 дня

### День 1

#### Этап 1. Быстрый старт и согласование

Время: 1-2 часа.

Задачи:

- выбрать стек frontend и backend;
- выбрать API для данных и новостей;
- определить структуру JSON-ответа;
- договориться о названиях endpoint и полей;
- создать базовую структуру проекта.

Результат:

- команда понимает, кто что делает;
- frontend и backend могут разрабатываться параллельно.

#### Этап 2. Backend MVP

Время: 3-5 часов.

Задачи:

- создать endpoint анализа тикера;
- получить базовые данные акции;
- получить новости;
- вернуть тестовый JSON;
- добавить обработку ошибок.

Результат:

- backend отдает данные хотя бы для 2-3 популярных тикеров.

#### Этап 3. Frontend MVP

Время: 3-5 часов.

Задачи:

- сделать форму поиска;
- сверстать основные блоки результата;
- подключить тестовый JSON или реальный backend;
- добавить loader и ошибки.

Результат:

- пользовательский сценарий уже можно показать: ввод тикера -> отображение результата.

#### Этап 4. AI-прототип

Время: 2-3 часа.

Задачи:

- подготовить prompt;
- протестировать AI-анализ на 2-3 тикерах;
- определить формат ответа AI;
- добавить fallback на случай ошибки AI API.

Результат:

- понятно, как будет формироваться итоговый отчет.

### День 2

#### Этап 5. Интеграция

Время: 2-4 часа.

Задачи:

- соединить frontend и backend;
- проверить реальные ответы API;
- исправить несовпадения по структуре данных;
- убедиться, что ошибки отображаются корректно.

Результат:

- приложение работает целиком.

#### Этап 6. Улучшение качества

Время: 2-4 часа.

Задачи:

- улучшить внешний вид интерфейса;
- добавить понятные тексты ошибок;
- ограничить количество новостей;
- улучшить AI prompt;
- убрать лишние данные из ответа.

Результат:

- приложение выглядит аккуратно и не ощущается сырым.

#### Этап 7. Тестирование

Время: 2 часа.

Проверить сценарии:

- корректный тикер: `AAPL`;
- корректный тикер: `TSLA`;
- корректный тикер: `MSFT`;
- пустой ввод;
- несуществующий тикер;
- ошибка внешнего API;
- ошибка AI API.

Результат:

- известны слабые места;
- критичные ошибки исправлены.

#### Этап 8. Подготовка к сдаче

Время: 1-2 часа.

Задачи:

- обновить README;
- подготовить краткое описание проекта;
- подготовить демонстрационные примеры;
- проверить запуск проекта с нуля;
- зафиксировать, какие ограничения есть у MVP.

Результат:

- проект готов к демонстрации.

## 7. Приоритеты

### Обязательно сделать

- ввод тикера;
- получение данных об акции;
- отображение информации о компании;
- отображение новостей;
- итоговый аналитический блок;
- обработка ошибок.

### Желательно сделать

- анализ влияния каждой новости;
- ссылки на источники новостей;
- аккуратный адаптивный интерфейс;
- сохранение последнего найденного тикера.

### Можно не делать в MVP

- авторизацию;
- личный кабинет;
- портфель пользователя;
- историю запросов;
- сложные графики;
- покупку или продажу активов;
- полноценные инвестиционные рекомендации.

---

## 8. Критерии готовности

Проект можно считать готовым к демонстрации, если:

- приложение запускается локально;
- пользователь может ввести тикер и получить результат;
- данные акции отображаются без критичных ошибок;
- новости отображаются в читаемом виде;
- AI-анализ формируется или корректно заменяется fallback-сообщением;
- некорректный ввод не ломает приложение;
- README объясняет идею, функциональность, запуск и ограничения проекта.

---

## 9. Ограничения MVP

- Анализ не является финансовой рекомендацией.
- Точность данных зависит от выбранных внешних API.
- Бесплатные API могут иметь лимиты запросов.
- Новости и рыночные данные могут обновляться с задержкой.
- AI-анализ может ошибаться и должен восприниматься как вспомогательная аналитика.
